Purpose: Close RTUX-03 so dev-review can converge on approved code without a human in the middle, while remaining an opt-in feature.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| verifier output → execute prompt | The reviewer (opus or codex) produces the verdict JSON. On retry, its `issues[]` content is inlined into the next execute prompt. An adversarial or buggy verifier could inject prompt-level instructions. |
| env var → loop budget | `REVISE_LOOP_MAX` and the CLI flag control how many extra agent invocations the script will make. Unbounded values mean unbounded cost. |
| `.verdict-normalized.json` (short-lived file) → feedback arg | File lives in `$RUN_DIR`, swept by `cleanup_runtime_artifacts` after the run. Not sensitive itself but is the channel by which reviewer-authored text reaches the executor. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering / Elevation | retry-loop phase-name → `phase_is_writable` | mitigate | `phase_is_writable` regex is anchored (`^execute-[0-9]+$`) so `execute-;rm-rf` or `verify-99` cannot slip through. Unit-tested in Task 2 acceptance. |
| T-02-02 | Injection (via reviewer content) | reviewer `issues[].description` → execute prompt | mitigate | `build_issues_list_markdown` renders via `jq -r` which safely escapes verdict fields; rendered lines are passed as a single `{ISSUES_LIST}` substitution, not evaled. The executor agent is still expected to follow the plan's "Do NOT deviate" rule (template language), which is a defense-in-depth — not the primary mitigation. |
| T-02-03 | Denial of Service / Cost | `REVISE_LOOP_MAX` value | mitigate | Option parser rejects non-integers; integer is treated as max-extra-passes so total budget = `N+1`. Default `0` preserves current behavior. Project README / help text documents that each pass costs real compute. |
| T-02-04 | Information Disclosure | reviewer `summary` / `issues` leaked into execute prompt | accept | The verifier runs on the same machine as the executor and the summary is the whole point of the feedback channel. No cross-tenant concerns in local CLI. |
| T-02-05 | Repudiation | numbered phase names in `state.json` | mitigate | Each retry pass writes its own `write_state_phase` entry with start + end timestamps and status. Existing Phase 8 eval scorer will see the per-pass record rather than a silent overwrite. |

| T-02-07 | Spoofing | env var vs CLI flag precedence | accept | CLI flag wins by virtue of running after default initialization. Documented in action notes. |
