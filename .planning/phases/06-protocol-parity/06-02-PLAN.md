<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Bounce pass output → persistent artifact | Plan content (possibly from an LLM, possibly containing untrusted text) is written to disk |
| Run directory path → find command | `$RUN_DIR` derived from repo root + timestamp; used in find/mkdir/cp |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Tampering | `cp "$output_file" "$RUN_DIR/outputs/bounce-${pass_padded}.txt"` | accept | `$output_file` at this point has passed `strip_human_summary` + size/structure validation; `$RUN_DIR` is derived from `${REPO_ROOT}/runs/dev-review-${TIMESTAMP}` (no user input); `$pass_padded` is derived from a bash loop counter (integer). No path traversal surface. |
| T-06-07 | Information Disclosure | `outputs/bounce-NN.txt` on disk | accept | Consistent with existing runtime artifact exposure (`run.log`, `pass-*-raw.md`). No new sensitive data class introduced. |
| T-06-08 | Denial of Service | `find "$outputs_dir" -maxdepth 1 -type f -name 'bounce-*.txt'` | mitigate | `-maxdepth 1` bounds traversal cost; glob is narrow; expected N ≤ MAX_BOUNCES (capped at 6 per dev-review.sh:787). |
| T-06-09 | Repudiation | Structural signal as ground truth | mitigate | Log line `Bounce artifacts: N pass file(s) in outputs/` is written to `run.log` alongside existing marker counts, creating an auditable record. |
</threat_model>
