<platform_detection_spec>
Per CONTEXT.md decisions (locked):
```bash
is_windows_host() {
  # Priority order — first match wins. Returns "true"/"false" on stdout.
  # 1. MSYS/Cygwin shell indicator
  if [[ "${OSTYPE:-}" == msys* || "${OSTYPE:-}" == cygwin* ]]; then
    printf '%s' "true"; return 0
  fi
  # 2. Windows Terminal binary present
  if command -v wt.exe >/dev/null 2>&1; then
    printf '%s' "true"; return 0
  fi
  # 3. cmd.exe present (covers WSL and bare Windows shells)
  if command -v cmd.exe >/dev/null 2>&1; then
    printf '%s' "true"; return 0
  fi
  printf '%s' "false"; return 0
}
```
</platform_detection_spec>

<launcher_strategy>
Per CONTEXT.md "Recommended: tail-window approach" (locked by decisions block):

- Main runner keeps inline invocation (zero IPC, zero exit-code forwarding).
- When LIVE_MODE=true AND is_windows_host returns "true": launch a detached tail window BEFORE the phase invocation, so the user sees streaming stderr as the agent runs.
- Prefer `wt.exe new-tab --title "<phase>" bash -c 'tail -f "<stderr_file>"'` (backgrounded with `&` + `disown` so the main shell does not block).
- Fallback: `cmd.exe /c start "<phase>" bash -c 'tail -f "<stderr_file>"'` (also backgrounded).
- The stderr file may not yet exist at launch time — pre-touch it (`: > "$stderr_file"` or `touch`) before calling the launcher so tail has something to follow. (Existing phase code writes to that path via `2>"$stderr_file"`, which truncates on first write, so pre-touching is safe.)
- Any non-zero exit from the backgrounded launcher is swallowed (`|| true`) and a WARNING logged. Main phase MUST continue regardless.
</launcher_strategy>

       ```bash
       # RTUX-01: Launch a visible tail window for the given phase's stderr file.
       # No-op when LIVE_MODE is not "true". Always returns 0 — failures log a
       # warning but never block the main phase (must-not-break invariant).
       maybe_launch_live_window() {
         local phase_name="${1:?maybe_launch_live_window requires a phase name}"
         local stderr_file="${2:?maybe_launch_live_window requires a stderr file path}"

         [[ "${LIVE_MODE:-false}" == "true" ]] || return 0

         if [[ "$(is_windows_host)" != "true" ]]; then
           if [[ "${LIVE_MODE_WARNING_LOGGED}" != "true" ]]; then
             log "WARNING: --live is Windows-only (OSTYPE=${OSTYPE:-unknown}); falling back to inline execution."
             LIVE_MODE_WARNING_LOGGED=true
           fi
           return 0
         fi

         # Pre-touch the stderr file so tail -f has something to follow.
         # The phase's own 2>"$stderr_file" redirection will truncate on first write.
         : > "$stderr_file" 2>/dev/null || true

         local title="phase:${phase_name}"
         local tail_cmd
         printf -v tail_cmd 'tail -f %q' "$stderr_file"

         # Preferred: Windows Terminal new-tab. Fall back to cmd.exe /c start.
         # Both are backgrounded + disowned so we do not wait on them.
         if command -v wt.exe >/dev/null 2>&1; then
           ( wt.exe new-tab --title "$title" bash -c "$tail_cmd" >/dev/null 2>&1 & disown ) 2>/dev/null \
             && return 0
           log "WARNING: wt.exe launch failed for phase '${phase_name}'; trying cmd.exe fallback."
         fi

         if command -v cmd.exe >/dev/null 2>&1; then
           ( cmd.exe /c start "$title" bash -c "$tail_cmd" >/dev/null 2>&1 & disown ) 2>/dev/null \
             && return 0
           log "WARNING: cmd.exe live-window launch failed for phase '${phase_name}'; continuing inline."
         else
           log "WARNING: no live-window launcher available for phase '${phase_name}'; continuing inline."
         fi

         return 0
       }
       ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user → CLI | Untrusted flag values enter the runner (`--live` is boolean-only, no argument — limited surface). |
| CLI → external launcher | `wt.exe` / `cmd.exe` process creation crosses the bash → Windows boundary. |
| background tail → filesystem | A new OS-level process reads the stderr_file in a separate window. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `phase_name` arg to maybe_launch_live_window | mitigate | `phase_name` is only used as a window title (not as a path or command argument that spawns a shell). All call sites pass literal string or `bounce-${pass_padded}` (integer-derived). Still, we never interpolate it into `bash -c` in a way that would allow command injection — the launcher uses `--title "$phase_name"` as an argv item, not as shell-interpolated text. |
| T-03-02 | Injection | `stderr_file` path passed to `tail -f` inside `bash -c` | mitigate | Use `printf -v tail_cmd 'tail -f %q' "$stderr_file"` so the path is bash-quoted; stderr_file values come only from trusted internal code (`$RUN_DIR/*-stderr.log`), not user input. |
| T-03-03 | DoS | Multiple orphaned `tail -f` processes accumulate across many runs | accept | User-observable; tail processes die when their window closes or when the stderr file is deleted/renamed. Documented in CONTEXT.md ("Window stays open; user closes manually"). Acceptable for a solo-dev workflow tool. |
| T-03-04 | Denial (self-DoS) | Launcher hang blocks main phase | mitigate | Launcher is backgrounded + disowned (`& disown`); main shell never waits on it. On failure, we `|| true` and log WARNING — main phase always proceeds. Asserted by Scenario C smoke test (state.json byte-equality). |
| T-03-05 | Info disclosure | stderr file contents exposed in a visible window | accept | That is the feature. No new channel is opened; stderr_file contents are already written to disk in the existing run directory. |
| T-03-06 | Privilege | Launcher attempts elevation or unexpected tool use | mitigate | Only two binary names invoked: `wt.exe` and `cmd.exe`, both from standard Windows paths. We do not write to the registry, env-persistent state, or start services. |
